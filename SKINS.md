# Skins (Personalities) — Contract

A **skin** is the *paint layer* of the kit: one CSS file in `personalities/<name>.css` that
defines a complete visual identity **purely by overriding design tokens** — with **zero changes
to structure or markup**. Same HTML, same layout, different look.

This doc is the contract every skin must satisfy. It applies equally to skins generated by
`scaffold/skin-from-photo.js` and any hand-built skin. If a change can't be expressed as a token
override (plus a few signature flourishes), it does **not** belong in a skin.

---

## Structure vs. Skin

The whole kit is a multiplication of two orthogonal layers:

| Layer | Lives in | Controls | In the builder |
|---|---|---|---|
| **Structure** | `sections/<slot>/<variant>` | *layout* — where things sit, what shape a section is | per-section variant buttons |
| **Skin** | `personalities/<name>.css` | *finish* — color, type, shape, depth, rhythm | the one global Skin dropdown |

A skin is **global and section-agnostic**: it paints every section identically and may **not**
assume which variants are on the page. `structure × skin` is what makes two sample sites look
unrelated.

---

## The five dimensions a skin sets

Every skin is a complete answer across the same token groups. Tokens are declared with defaults in
`base/base.css`; a skin overrides them in `:root`.

1. **Color** — the semantic color tokens (roles, never raw hex in sections):
   `--c-bg`, `--c-bg-alt`, `--c-bg-deep`, `--c-surface`,
   `--c-ink`, `--c-ink-soft`, `--c-ink-invert`, `--c-ink-invert-soft`,
   `--c-brand`, `--c-brand-deep`, `--c-brand-ink`, `--c-accent`,
   `--c-line`, `--c-line-strong`.
2. **Type** — `--font-display`, `--font-body` (with the matching `@import`), plus
   `--fw-*` weights, `--tracking-*`, `--leading-tight`.
3. **Shape** — `--radius`, `--radius-lg`, `--radius-pill`, `--border-weight`, `--rule-weight`,
   and the button shape tokens `--btn-radius`, `--btn-transform`, `--btn-weight`, `--btn-tracking`.
4. **Elevation** — `--shadow`, `--shadow-lg`, `--card-border`.
5. **Density / rhythm** *(optional)* — `--section-pad`, `--gap` when the skin wants a tighter or
   airier feel.

…plus a small set of **signature flourishes**: a *handful* of targeted CSS rules that give the skin
its unmistakable tell. Keep these few and generic to component classes the kit already defines
(`.eyebrow`, `.card`, `.section-head`, `h1/h2`, `.btn--brand`). Examples already shipped:

- **Industrial Bold** — uppercases all headings; offset-block (`translate(-3px,-3px)` + hard
  shadow) card hover; brand-on-ink eyebrow chip.
- **Warm Residential** — italic serif `<em>` emphasis in headlines; gentle `translateY` hovers;
  sage eyebrow.

---

## What a skin is NOT

- **Not layout.** It never adds, removes, reorders, or restructures sections. That's variants.
- **Not markup.** Sections never receive skin-specific HTML or skin-specific class names.
- **Not section internals.** Flourishes target shared component classes, never one variant's
  private structure.
- **Not color-picked.** Color is *derived*, never chosen by hand — photo → palette → WCAG-corrected
  → paired with a form preset (`skin-from-photo.js`). The reviewer approves finished looks; the
  workflow never asks "what color?".

---

## The contract checklist

A skin is valid only if **all** of these hold:

- [ ] Touches **only** `:root` token overrides + a few signature flourishes — no section internals.
- [ ] Defines a value for **every token in all five dimensions** (a complete look, not a partial tint).
- [ ] Looks right on **any** combination of section variants (test via Shuffle in the builder).
- [ ] All text/background pairings pass **WCAG AA** contrast (the generator enforces this; verify by
      hand for hand-built skins). Key pairs: ink-on-bg, ink-on-surface, brand-ink-on-brand,
      ink-invert-on-bg-deep.
- [ ] Both display **and** body fonts are loaded via the skin's own `@import`.
- [ ] Flourishes reference shared component classes only (`.eyebrow`, `.card`, `.section-head`,
      `h1/h2`, `.btn--*`) — never a variant-specific selector.
- [ ] Named by **feel**, not by trade (e.g. `warm-residential`, not `florist`). A suggested-industry
      note may live in the header comment, but the name stays trade-agnostic — same rule as variant
      naming.
- [ ] Registered in `demo/index.html` (`SKINS[]` array + a `<link>` for the file) so it shows in the
      builder dropdown.

## File skeleton

```css
/* ============================================================
   PERSONALITY — <Display Name>
   For: <suggested industries — a hint, not a lock>

   Tells: <one line describing the signature look>
   ============================================================ */
@import url('https://fonts.googleapis.com/css2?family=<Display>&family=<Body>&display=swap');

:root {
  /* Color */      /* ...the 14 --c-* tokens... */
  /* Type */       /* --font-display, --font-body, --fw-*, --tracking-*, --leading-tight */
  /* Shape */      /* --radius*, --border-weight, --rule-weight, --btn-* */
  /* Elevation */  /* --shadow, --shadow-lg, --card-border */
  /* Rhythm */     /* --section-pad, --gap  (optional) */
}

/* Signature flourishes — a few rules, shared classes only */
```

## Adding a skin

1. **Generate** (preferred): `node scaffold/skin-from-photo.js <image> <skin-name> --form <preset>`
   (presets: `modern`, `elegant`, `industrial`, `clean`). The generator derives the palette and
   WCAG-corrects it. Or hand-build from the skeleton above.
2. **Register** it in `demo/index.html`: add to the `SKINS[]` array and add a `<link>`.
3. **Verify** against the checklist — Shuffle through variants in the builder, confirm contrast.
4. **Update** the inventory count in memory (`project_kit_variant_inventory.md`).

See the differentiation ledger for how skins combine with variants so no two shipped templates
overlap too much.
